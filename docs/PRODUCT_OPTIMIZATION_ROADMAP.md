# TheLoom 产品优化路线图

状态：提案

基线：`v0.33.0 / R16`

目标：在不继续扩张平行模块的前提下，把现有能力收束为两条可长期使用的主工作流：

- 小说作者：规划 → 写作 → 修订 → 投稿交付
- 独立游戏开发者：编剧 → 分支实现 → 自动验证 → 引擎集成

## 排序原则

1. 先统一数据主轴，再增加工作台和导出能力。
2. 先解决会造成重复维护、数据失真和无法交付的问题。
3. 每个批次必须包含旧项目迁移、文件夹往返、撤销恢复和自动化验证。
4. 新视图优先作为现有权威数据的投影，不再创建平行正文或平行场景记录。
5. 引擎侧以稳定 ID 和版本化协议为准，不依赖标题、显示顺序或中文文案。
6. 旧示例文件夹仅作为迁移回归数据，不反向约束新结构；全部功能改造完成后统一重做正式示例。

## R17：统一叙事主轴

### R17-0 基线与样例固化

| 项目 | 内容 |
|---|---|
| 范围 | 固化一份长篇小说样例、一份多分支解谜样例和一份从旧版本升级的项目样例 |
| 自动化 | 覆盖导入、保存、重新加载、撤销、项目快照、文件夹 Markdown 往返和引擎包生成 |
| 验收 | 后续所有数据迁移都能证明三份样例内容、顺序、引用和技术名未丢失 |

### R17-1 卷章场景结构化

推荐采用渐进方案：继续使用 `Document` 作为场景正文，给文档文件夹增加可选结构角色，而不是再创建一套平行正文模型。

| 项目 | 内容 |
|---|---|
| 数据 | 文档文件夹可标记为卷、章、小节或普通分组；旧文件夹默认保持普通分组 |
| 规则 | 卷可包含章，章可包含场景；非法层级只提示，不破坏已有内容 |
| 界面 | 新建卷、新建章、新建场景成为明确动作；普通文件夹仍保留 |
| 编译 | 章节编译、连续稿、卡片墙、登场统计和节奏图统一读取结构角色 |
| 验收 | 文件夹改名不改变结构含义；移动整章后所有场景顺序与编译结果一致 |

### R17-2 场景权威主轴

| 项目 | 内容 |
|---|---|
| 权威对象 | `Document` 继续作为场景的正文与元数据权威来源 |
| 引用 | 大纲行可关联章或场景；时间线事件可关联一个或多个场景；流程继续稳定关联场景 |
| 已有能力 | 角色弧线和伏笔台账沿用现有文档引用，不创建第二套场景 |
| 删除策略 | 删除场景前列出跨模块引用；确认后解除引用或连带删除投影记录 |
| 验收 | 场景标题、状态、POV、地点、故事时间和顺序只需修改一次 |

### R17-3 联动投影视图

| 项目 | 内容 |
|---|---|
| 大纲 | 已关联行展示场景实时标题和状态；计划文字仍允许独立存在 |
| 时间线 | 事件显示关联场景，支持从时间线直达正文 |
| 规划 | 卡片墙、节奏图、登场统计全部按统一章场顺序工作 |
| 导航 | 增加返回上一个位置和最近访问对象，降低跨模块跳转成本 |
| 验收 | 重命名、移动或归档场景后，不出现失效标题副本和静默悬挂引用 |

### R17-4 项目工作区预设

| 预设 | 默认显示 | 收起到更多 |
|---|---|---|
| 小说 | 文档、实体、规划、资料、大纲 | 流程、变量、资源、地图、时间线、风暴 |
| 互动叙事 | 流程、文档、实体、变量、资源 | 规划、大纲、时间线、地图、资料、风暴 |
| 通用 | 保持当前完整导航 | 无 |

预设只控制信息架构和默认术语，不裁剪功能，不改变项目数据。

## R18：小说生产闭环

### R18-1 长篇结构操作

- 场景拆分：在光标或块边界处拆成两个场景，保留修订信息和引用提示。
- 场景合并：选择相邻场景合并，冲突元数据先预览。
- 卷章重排：整卷、整章拖动，单次撤销。
- 自动编号：可选择中文数字、阿拉伯数字或不编号。
- 批量设定：按卷或章设置状态、修订轮次、目标字数和分类。

验收标准：任何拆分、合并和重排都可一次撤销；伏笔、弧线、流程和时间线引用不会静默丢失。

### R18-2 DOCX 成稿导出

| 项目 | 首批范围 |
|---|---|
| 结构 | 书名、作者、卷、章、场景分隔、分页 |
| 样式 | 标题样式、正文样式、首行缩进、行距、页边距 |
| 内容 | 可选择仅正文，或包含场景标题、批注摘要和修订信息 |
| 模板 | 至少提供投稿稿、编辑审阅稿两种预设 |
| 验证 | 导出后重新解析 DOCX，比对章节数、段落数、文本和顺序 |

暂不在首批实现复杂 Word 修订标记；先保证成稿结构、样式和文本可靠。

### R18-3 写作进度工作台

- 全书、卷、章、场景四级目标。
- 今日新增字符、最近七日趋势和当前修订轮次进度。
- 最近编辑、未完成批注、待回收伏笔和未达标章节入口。
- 字数口径可选：中文字符、含标点字符、英文单词、仅正文。

### R18-4 修订与校对闭环

- 把场景快照提升为可筛选的修订任务和差异列表。
- 支持逐项接受、保留或标记待议，不直接覆盖正文。
- 本地检查重复词、连续标点、全半角、角色称谓和专名一致性。
- AI 建议继续走提案、预检、确认、单次提交，不绕过现有安全边界。

## R19：游戏运行时语义

### R19-1 运行时事件协议 v2

每次运行时事件至少包含：

- `flowId`、`flowTechnicalName`
- `nodeId`、`nodeTechnicalName`、子流程路径
- 节点类型、标题、正文、说话人
- 自定义字段与附件资源 ID
- 进入事件、展示事件、离开事件
- 本步变量和实体属性的变化摘要
- 触发它的 `edgeId` 与选项稳定键

兼容要求：保留 v1 常用字段；引擎包和运行时明确声明协议版本；旧包有确定的降级行为。

### R19-2 跨流程调用

| 能力 | 语义 |
|---|---|
| entry | 一个流程可以声明多个稳定命名入口 |
| jump | 跳到目标流程入口，不返回 |
| call | 调用目标流程入口，完成后返回调用点 |
| return | 显式返回，可携带简单返回值 |
| 参数 | 首批支持布尔、数值、文本 |
| 存档 | 调用栈、入口和参数必须进入运行时快照 |

验收标准：编辑器演出、TypeScript 运行库和 Godot 运行库对同一包产生一致的节点序列、变量终值和返回位置。

### R19-3 外部事件节点

提供不直接执行引擎代码的声明式事件节点：

- 事件名使用稳定技术名。
- 参数使用模板化字段并进行类型检查。
- 运行时通过回调交给宿主引擎处理。
- 可配置立即继续、等待宿主确认或等待返回值。
- 编辑器演出允许为事件配置模拟响应。

### R19-4 场景化回归测试

- 保存测试名称、流程入口、初始变量、已访问节点和随机种子。
- 断言可达结局、变量终值、节点是否访问和事件是否触发。
- 批量运行并显示节点、边、结局和条件覆盖率。
- 流程修改后自动标出受影响或失败的测试。

### R19-5 流程编辑效率

- 节点与连线复制、粘贴、重复和跨流程复制。
- 多选对齐、等距分布、批量改模板和批量改字段。
- 从选区封装为剧情片段或独立可调用流程。
- 技术名安全重命名，同时更新条件、指令、seen/unseen 和引用索引。

## R20：引擎交付生产化

### R20-1 导出配置与基线

- 导出规则保存为命名配置，可选择流程、资源范围、布局和注释。
- 增量清单与配置绑定，并能随项目文件夹保存或显式导入导出。
- 导出前统一执行脚本检查、高级体检、路径测试和测试场景。

### R20-2 自包含引擎包

导出选项：

- 仅数据包。
- 数据包加被引用资源原文件。
- 数据包加目标运行库。
- 完整包加校验清单和许可证来源表。

验收标准：把完整包复制到一台没有 TheLoom 项目文件夹的机器，示例工程仍能加载、显示对白并读取附件。

### R20-3 CLI 与目录同步

- 提供无界面导出命令，输入项目目录和导出配置名。
- 支持写入固定 Godot/Unity 项目目录和仅输出变化文件。
- 非零退出码区分体检失败、测试失败、Schema 不兼容和文件写入失败。
- 给出 CI 示例和开发期监听模式。

### R20-4 官方引擎适配

1. 先让 Godot 运行库与 TypeScript 完整语义一致，消除脚本子集差异。
2. 再提供 Unity 最小包：数据加载、流程运行、选项 UI、存读档和事件回调。
3. Unreal、Ren'Py、Yarn/Ink 转换器进入后续候选，不阻塞首批生产闭环。

## R21：本地化与配音

### R21-1 稳定本地化键

- 对白、选项、场景说明和可本地化自定义字段拥有稳定 `locKey`。
- 修改原文不会改变键；复制内容默认生成新键。
- 导入旧项目时确定性生成键并提供冲突报告。

### R21-2 翻译工作台

- 按语言显示未翻译、待更新、已审校和已锁定状态。
- CSV、XLSX、JSON 往返，导入前预检。
- 检查缺失文本、变量占位符差异、长度溢出和孤立键。

### R21-3 配音生产状态

- 文本键关联角色、音频资源、录制状态、时长和备注。
- 统计角色台词量与预计时长。
- 引擎包按文本键输出对应音频资源引用。

## R22：正式示例与交付验收

### R22-1 《老伦敦寻人记》示例重制

- 旧示例文件夹继续保留为自动化迁移样例，不作为界面和数据模型设计依据。
- 以维多利亚时代老伦敦寻人谜案为主线，同时提供可连续阅读的小说稿与可游玩的分支解谜流程。
- 覆盖卷章场景、角色与地点、伏笔、故事时间线、线索变量、条件分支、失败结局和真结局。
- 提供章节编译、DOCX 成稿、完整项目备份及 Godot / Unity 引擎包，用于最终端到端验收。
- 示例内容在 R17-R21 的数据结构和导出协议稳定后一次重做，期间不反复维护临时样例。

## 每批统一完成标准

每个批次只有同时满足以下条件才进入下一项：

1. `normalizeProject` 能迁移旧项目，迁移重复执行不产生二次变化。
2. JSON 备份、项目槽位、撤销重做和版本快照可用。
3. 涉及桌面文件夹数据时，Markdown/frontmatter 与 Rust 读写无损往返。
4. 删除、移动、拆分、合并均检查跨模块引用。
5. `npm run build` 与 `npm test` 通过；涉及桌面存储时增加 `cargo test --lib`。
6. 可见交互完成浏览器实际点验，包括空状态、错误状态、取消和撤销。
7. README、帮助面板、示例项目和版本迁移说明同步更新。

## 暂缓项

以下内容不应插入 R17 至 R20 的主线：

- 新增更多独立规划模块。
- 多窗口和复杂协同编辑。
- 完整排版软件能力。
- 引擎内直接制作动画状态机或谜题逻辑。
- 在没有稳定本地化键之前铺开完整配音管理。

## 建议的第一个开发任务

从 R17-0 开始，随后只实现 R17-1 的最小闭环：

1. 为文档文件夹增加可选结构角色。
2. 在 `normalizeProject` 中迁移并校验层级。
3. 让连续稿与章节编译读取结构角色。
4. 补 JSON、Markdown/frontmatter 和旧项目升级测试。
5. 最后再调整 Navigator 的新建入口和显示文案。

这个顺序先稳定数据语义，再改变界面，能把后续场景联动、DOCX 导出和进度统计建立在同一套结构上。
